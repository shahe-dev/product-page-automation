# Extracting floor plan data from real estate brochures at scale

**No turnkey solution exists for this problem — but a hybrid multi-stage pipeline combining YOLO-based layout detection, template matching, and targeted VLM reasoning is the most reliable architecture proven in adjacent industries.** The core insight from this research is that the brittleness of the current PyMuPDF + pdfplumber + VLM stack stems from trying to solve fundamentally different sub-problems (page segmentation, image classification, data attribution, cross-page grouping) with a single undifferentiated approach. Production systems that handle similar document complexity in insurance, financial services, and construction all converge on the same pattern: decompose the problem into discrete stages, use specialized models for each stage, and route low-confidence results to human reviewers. For UAE developer brochures specifically, the lack of standardization across Emaar, DAMAC, Sobha, and Nakheel makes this decomposition not just helpful but essential.

## Why the current stack keeps breaking

The PyMuPDF + pdfplumber + VLM combination fails because it conflates two fundamentally different tasks: **structural understanding** (where are the floor plans on this page, and what text belongs to which plan) and **semantic extraction** (what do the numbers mean). PyMuPDF and pdfplumber extract raw text and images without understanding page layout — they can’t tell whether “1,250 sqft” belongs to the 2BR unit or the 1BR unit on the same page. The VLM is then asked to solve both attribution and extraction simultaneously from a full-page image, which is where hallucination and misattribution errors compound.

The “every fix breaks another brochure” pattern is a classic symptom of **rule-based extraction applied to visually diverse documents**. When extraction logic is written as conditional rules (e.g., “the sqft is always the third number on the page”), any layout variation creates a regression. The solution is to replace brittle positional rules with learned spatial understanding — models that detect regions regardless of where they appear on the page.

Research from UC Berkeley’s TWIX benchmark quantifies the problem: GPT-4 Vision achieves only **60–80% field-level accuracy** on complex multi-record documents, with attribution errors being the primary failure mode. For pages with multiple units, the VLM “occasionally selects the wrong one” when similar fields appear near each other.  This matches the exact failure pattern described.

## The five-stage architecture that works

The consensus architecture from production systems in insurance (HealthEdge, ABBYY), document AI (IBM Docling, MinerU), and enterprise extraction (Reducto) follows a clear pattern.  Each stage is independently testable, debuggable, and replaceable.

**Stage 1: PDF rendering and text extraction.** Convert each page to a high-resolution image (300 DPI) using PyMuPDF or pdf2image. Simultaneously extract the programmatic text layer with bounding box coordinates using pdfplumber or Docling’s PDF backend. This dual extraction — visual rendering plus text coordinates — provides the foundation for everything downstream. Born-digital PDFs (most Dubai developer brochures) yield clean text coordinates; scanned brochures fall back to OCR via EasyOCR or Tesseract.

**Stage 2: Layout detection with a fine-tuned YOLO model.** This is the highest-leverage intervention. A **DocLayout-YOLO** or **YOLOv11** model, fine-tuned on annotated brochure pages, detects and classifies every visual region on the page with bounding boxes. Define **5–6 custom classes**: `floor_plan` (individual unit layout), `position_diagram` (unit location in building), `data_table` (specifications), `building_render` (3D visualization), `lifestyle_photo`, and `site_plan`. YOLO models inherently handle **multiple instances of the same class per page** — the exact capability needed when three floor plans appear on one page. Each gets its own bounding box with a confidence score. DocLayout-YOLO’s Global-to-Local Controllable Receptive Module specifically handles elements at different scales on the same page.  Fine-tuning requires **300–500 annotated pages** (achievable in 2–3 weeks using CVAT or Label Studio with images from 50–100 brochures across major developers), and training converges in hours on a single GPU.

**Stage 3: Per-region extraction and attribution.** Once regions are detected and cropped, each floor plan image is paired with nearby text using **spatial proximity analysis**. Text elements within or adjacent to a floor plan’s bounding box are associated with that plan. This is where the SPADE approach (Spatial Dependency Parsing) proves valuable — it models the document as a graph where text tokens and visual regions are nodes, and spatial distance determines edge weights.  For tables containing unit specifications, Docling’s **TableFormer** model recognizes complex table structures   (including borderless tables and merged cells) at **97.9% accuracy**, dramatically outperforming general-purpose tools. The attributed data for each unit — type, bedroom count, area, floor range — is extracted from its associated text regions and table rows.

**Stage 4: VLM verification and edge case handling.** A vision-language model processes only the cases where Stage 3 produces low-confidence results or ambiguous attributions. Instead of sending full pages, send **individual cropped regions with surrounding context** — this dramatically improves accuracy by constraining the VLM’s attention. Gemini 2.0 Flash is the cost leader at **$0.10/M input tokens** (25× cheaper than GPT-4o), making it viable for routine verification. Claude 3.5 Sonnet has the **lowest hallucination rate (0.09%)** among commercial VLMs, making it preferable for numerical data where fabrication is unacceptable. For self-hosted deployments, **Qwen2.5-VL-7B** runs on a single GPU and supports bounding box output  — a unique capability among open-source VLMs.

**Stage 5: Validation, QA, and human review.** Apply cross-field validation rules: a 1BR unit should be 500–900 sqft, a 2BR 900–1,500 sqft, a 3BR 1,300–2,500 sqft. Floor numbers must fall within the building’s stated height. Total units extracted should match the brochure’s stated count. Every field gets a confidence score based on extraction method (template match → high, VLM extraction → medium, spatial inference → lower). Fields below a **0.85 confidence threshold** route to a human review queue with a side-by-side UI showing the source PDF page with highlighted regions alongside the extracted data. 

## Solving each specific failure mode

**Multiple numbers on a page (misattribution).** The YOLO detection model creates semantic zones — each floor plan gets a bounding box that defines its spatial territory. Text extraction within each zone links numbers to their correct unit. For specification tables, TableFormer preserves row-column structure so “1,250 sqft” stays associated with “2BR Type A” in the same row. The SPADE graph-parsing approach provides a second layer: it computes spatial dependency scores between every text token and every detected region, resolving cases where a number sits between two floor plans. The key insight is that **deterministic spatial reasoning should handle 80–90% of attribution**, with VLM reasoning reserved for truly ambiguous layouts. 

**Multiple floor plans on a single page.** This is precisely what object detection models are designed to solve. YOLO’s non-maximum suppression preserves separate detections for distinct floor plans while preventing duplicate detections of the same plan. A model trained on Dubai developer brochures will learn that Emaar’s typical 2-up layout places plans side by side, while DAMAC often stacks them vertically. The bounding box for each detected floor plan provides the crop coordinates directly — no VLM-based image cropping needed.

**Multi-floor villas spanning multiple pages.** Implement a **cross-page grouping stage** that runs after per-page extraction. The grouping algorithm uses three signals: (1) page sequence — consecutive pages with floor plans of the same unit type, (2) heading text — OCR detection of “Ground Floor,” “First Floor,” “Roof Terrace” labels, and (3) unit identifier consistency — the same “Villa Type A” label appearing across pages. For ambiguous cases, a VLM prompt (“Do these two floor plans belong to the same residential unit?”) provides high-accuracy confirmation. Many brochures include a table of contents that maps unit types to page ranges — extracting this index page first provides a structural skeleton for the entire brochure.

**Building position diagrams misidentified as floor plans.** These two image types have **distinct visual signatures** that a fine-tuned classifier can learn. Floor plans show high line density (walls, doors, windows), room labels, furniture symbols, and dimension annotations.   Position diagrams show a simplified building outline with one unit highlighted in a contrasting color, minimal interior detail, and no room labels. A lightweight **EfficientNet-B0 classifier** trained on 500–1,000 labeled examples from brochures achieves reliable separation.  For rapid prototyping before collecting training data, **CLIP zero-shot classification** with descriptive text prompts  (“a detailed architectural floor plan showing room layout” vs. “a building diagram showing apartment location”) provides a reasonable starting baseline.

**Unit type variants not detected (2BR Type A vs. 2BR Type B).** This is primarily a text extraction and grouping problem, not a vision problem. Dubai developer brochures typically label variants explicitly (“Type A,” “Type B”) in headings or nearby text. The extraction pipeline should: (1) detect all variant labels via OCR/text extraction, (2) associate each label with its nearest floor plan via spatial proximity, and (3) generate unique identifiers combining bedroom count and variant label. When variants appear on the same page, the YOLO detector segments each floor plan independently, and the attributed text provides the variant designation.

## Template matching plus AI creates the most robust system

The hybrid approach outperforms both pure template-based and pure AI-based extraction. UC Berkeley’s TWIX research demonstrates that for templatized documents, template-based extraction is **520× faster and 3,700× cheaper** than VLM baselines.   Since major Dubai developers like Emaar tend to reuse brochure templates across projects (with different unit details), **building templates for the top 10–15 developers covers 60–70% of volume** at near-zero marginal cost.

The practical architecture works as follows. When a brochure arrives, the system first attempts **developer identification** (from logos, headers, or metadata) and template matching. Known templates use deterministic extraction rules that produce near-100% accuracy. Unknown formats or low-confidence template matches fall through to the AI pipeline (YOLO detection → spatial attribution → VLM verification). Critically, **every human-validated AI extraction becomes training data for new templates** — creating a flywheel where the system gets faster and cheaper over time. 

This hybrid approach directly solves the “every fix breaks another brochure” problem. Template rules are scoped per developer — changes to the Emaar template cannot affect DAMAC extraction. The AI fallback generalizes across formats without developer-specific rules.  And the regression test suite (50–100 diverse brochures with verified ground truth) catches any degradation before deployment.

## The tools worth building on

**Docling (IBM)** is the strongest open-source foundation for the overall pipeline,  with 37,000+ GitHub stars and active development under the Linux Foundation. Its sequential architecture (PDF parsing → OCR → layout analysis → table recognition → assembly) provides a clean skeleton to extend with custom stages.   Its TableFormer model handles the specification table extraction that other tools struggle with.

**DocLayout-YOLO** or **YOLOv11** should be the layout detection backbone, fine-tuned on annotated brochure pages. The Ultralytics ecosystem provides well-documented fine-tuning pipelines,  and inference runs at real-time speed on modest hardware. For the 3,000–15,000 pages per month this pipeline needs to handle, a single GPU server is more than sufficient.

For VLM inference, the cost-optimal strategy is **Gemini 2.0 Flash for bulk verification** (~$3–5/month for 300 brochures) with **Claude as fallback for high-stakes numerical extraction**. For organizations requiring data residency or wanting to eliminate API costs, **Qwen2.5-VL-7B** self-hosted on a single H100 or A100 provides competitive accuracy with full data control.

**RasterScan** deserves mention as the most production-ready API for converting extracted floor plan images into structured room geometry (walls, doors, room boundaries), though this level of floor plan interpretation may exceed the current requirements of extracting metadata rather than spatial structure.

No UAE-based proptech company currently addresses this specific brochure-to-data problem — this represents a clear market gap in the Dubai ecosystem.

## Building the QA layer that makes it production-grade

The difference between a prototype and a production system is the validation layer. The recommended QA architecture has four tiers operating in sequence.

**Tier 1: Automated field validation.** Every extracted record passes through domain-specific rules — area ranges by bedroom count, floor number bounds, required field completeness checks, and unit count reconciliation against the brochure’s stated total. These rules catch approximately **40–50% of extraction errors** with zero human effort.

**Tier 2: Statistical anomaly detection.** Compare extracted values against distributions from previously processed brochures. Flag sqft values more than two standard deviations from the mean for their unit type. Detect duplicate extractions and missing unit variants. This catches another **15–20% of errors**, particularly systematic failures where the pipeline misinterprets a page structure.

**Tier 3: Confidence-routed human review.** Records where any field falls below the 0.85 confidence threshold enter a review queue. The review UI displays the source PDF page with color-coded bounding boxes alongside the extracted structured data, with click-through from each field to its source location on the page.  Tracking reviewer override rates by pipeline stage reveals which stages need improvement — **the most actionable metric for pipeline development**. 

**Tier 4: Regression testing.** Maintain a golden test suite of 50–100 brochures stratified by developer, unit type (apartment, villa, townhouse), and page complexity. Every pipeline change runs against this suite with automated precision/recall/F1 scoring per field. CI/CD gates block deployments that degrade accuracy below thresholds on any developer. This directly prevents the “fix one, break another” pattern that plagues the current system.

## Conclusion

The path forward requires abandoning the monolithic VLM-first approach in favor of a decomposed pipeline where each failure mode is addressed by a purpose-built stage. The highest-leverage first step is **annotating 300–500 brochure pages and fine-tuning a YOLO model for floor plan detection** — this single intervention solves the page segmentation and image cropping problems that no amount of VLM prompting can reliably address. The second priority is implementing **spatial attribution logic** (SPADE-style graph parsing or simpler proximity-based association) to correctly link numbers to floor plans without relying on VLM reasoning. Template matching for known developers provides the cost efficiency and reliability needed at scale, while VLM fallback handles the long tail of new formats.  The regression test suite and confidence-routed human review close the quality gap. Together, these components create a system that improves with every brochure processed rather than breaking with each new format encountered.
