Critical: Images never reach Google Drive
This is a functional regression from v3. Tracing the image lifecycle end-to-end:
Extraction pipeline: Steps 1-8 extract, classify, optimize images → Step 8 packages into ZIP → Materialize step extracts images from ZIP and uploads to GCS at materials/{project_id}/images/. Pipeline ends.
Generation pipeline: Loads JSON metadata from GCS → Generates content → Populates sheet → _step_upload_cloud_generation moves only the sheet to Drive → Done.
Nobody uploads the actual images to Drive. In v3, step 13 creates the full Drive folder structure (Project Folder/Images/{category}/). In the revised plan, that step was replaced by a generation-specific version that explicitly says "Images are already in GCS" and skips them.
Your team works with these files in Drive daily — your own notes flag Drive upload as a core part of the workflow. GCS-only storage breaks that.
Fix: Add a Drive upload step at the end of the extraction pipeline, between materialize and job completion. This uploads shared assets (images, PDF source, floor plan data) to Drive once. Store the drive_folder_id on the MaterialPackage or Project record. Generation jobs then reference this existing Drive folder when placing their template-specific outputs (sheets). This mirrors the v3 structure but only runs the heavy upload once instead of per-template.
The step configuration would become 12 extraction steps (1-10 + materialize + upload_shared_assets), and generation's upload step just moves the sheet into the existing project folder.

Medium: create_and_dispatch_job not updated
Task B.6 calls self.create_and_dispatch_job(user_id=..., job_type=JobType.GENERATION, material_package_id=...). Task B.5 updates JobRepository.create_job() to accept these parameters. But create_and_dispatch_job is a JobManager method that wraps the repository call plus Cloud Task dispatch — it's the intermediary, and it's never explicitly updated to accept and pass through job_type and material_package_id. The call in B.6 would fail at runtime.
Fix: Add an explicit task (B.5b or fold into B.5) that updates JobManager.create_and_dispatch_job() to accept job_type: JobType = JobType.FULL and material_package_id: Optional[UUID] = None and forward them to the repository.

Medium: Process route (B.10) doesn't follow existing patterns
The /process/extract endpoint instantiates services inline (StorageService(), JobManager(job_repo, task_queue), TaskQueue()). The existing upload routes almost certainly use FastAPI dependency injection. This should follow whatever pattern upload.py uses — probably Depends() for DB session, storage service, and job manager. The route also uses request: Request in the function signature but doesn't import Request from FastAPI.
Fix: When implementing B.10, mirror the dependency injection pattern from the existing backend/app/api/routes/upload.py rather than using the inline instantiation shown in the plan.

Lower priority items
Context mapping in B.4 (_step_load_material_package): The method maps extracted_text.pages to ctx["extraction"]["text_by_page"]. Verify this matches what _step_generate_content actually reads from the pipeline context. Content generation likely reads from ctx["structured_data"] (which is correctly mapped), but if there's a fallback path that touches extraction text, the key structure needs to match v3's format exactly.
Race condition in B.8 (_step_finalize_generation): The check-then-create pattern for GenerationRun isn't atomic. Two simultaneous generation jobs for the same template would both read existing_run=None, both try to INSERT, one hits the unique constraint. Low risk for MVP since you'd rarely dispatch duplicate template jobs, but consider using INSERT ... ON CONFLICT or wrapping in a transaction with SELECT FOR UPDATE.
Early project creation is a UX change: In v3, projects appear on the dashboard only after step 14 (fully complete). In the revised plan, the project is created during extraction in DRAFT status before generation even starts. If generation fails, users see a project with no content. The workflow_status=DRAFT flag handles this logically, but the frontend needs to display this state properly (C.7 partially addresses this with generation run status badges).
ZIP double-handling in materialize: Step 8 packages images into a ZIP, then the materialize step unzips them to upload individually to GCS. This is redundant I/O. A cleaner approach would be uploading optimized images to GCS directly during step 7 (optimize) and storing the GCS paths in the manifest. The ZIP would still be created for the download artifact but wouldn't be the source of truth for GCS persistence. Not a blocker but worth noting for when you optimize.
No TTL cleanup for MaterialPackages: expires_at is set to 30 days but nothing enforces it. GCS objects and DB records accumulate indefinitely. Add a Cloud Scheduler job or a periodic cleanup task to the backlog.
