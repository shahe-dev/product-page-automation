## Template Structures

### Common Column Format

All templates follow this column structure:

| Column | Content |
|--------|---------|
| A | Guidelines/Comments (internal notes, character limits) |
| B | Field Name (database/API identifier) |
| C | EN (English content) |
| D | AR (Arabic content) |
| E | RU (Russian content) |

### Core Sections (Present in All Templates)

| Section | Required Fields | Character Limits |
|---------|-----------------|------------------|
| **SEO** | Meta Title, Meta Description, URL Slug, Image Alt | 50-60, 150-160, -, 80-125 |
| **Hero** | H1, Description, Starting Price, Payment Plan, Handover | 30-60, 250-400, -, -, - |
| **About** | H2, Description, Selling Points (3-5) | 20-50, 400-700, 50-80 each |
| **Project Details** | Developer, Location, Property Types, Bedrooms, Price | Varies |
| **Amenities** | H2, 5 Amenities (Title + Description each) | 20-50, 20-40/100-150 |
| **Payment Plan** | H2, Description, Milestones (3) | 20-50, 400-800, - |
| **Location** | H2, Description, Nearby (3-4) | 20-50, 300-550, - |
| **Developer** | H2, Description | 20-50, 250-500 |
| **FAQ** | 3-5 Q&A pairs | 60-80/100-200 each |

## Integration with PDP System

### Database Schema Mapping

Templates map to database tables as follows:

```
Template Field          -> Database Column
-------------------------------------------
meta_title             -> projects.meta_title
meta_description       -> projects.meta_description
slug                   -> projects.slug
h1                     -> project_content.hero_h1
hero_description       -> project_content.hero_description
starting_price         -> projects.starting_price
payment_plan           -> projects.payment_plan_summary
handover               -> projects.handover_date
about_h2               -> project_content.about_heading
about_description      -> project_content.about_description
amenity_*_title        -> project_amenities.title
amenity_*_description  -> project_amenities.description
location_*             -> project_locations (related table)
developer_*            -> developers (related table)
faq_*                  -> project_faqs (related table)
```

### Content Generation Flow

```
1. User selects template type (OPR, MPP, ADOP, etc.)
2. System loads template structure from Sheets
3. AI generates content per field with character limits
4. Content populates Google Sheet (project-specific copy)
5. QA validates against template constraints
6. Approved content exports to website
```

### API Integration

Templates are accessed via Google Sheets API:

```python
# Template ID for .env
TEMPLATE_SHEET_ID = "1pef6Q-54l2mFOX6QgwOLQONviBijgaRgI7gA2GHn_Ck"
```

## Validation Rules

### Required Fields (All Templates)

These fields must have content before publishing:

- meta_title
- meta_description
- slug
- h1
- hero_description
- starting_price
- about_description
- At least 3 amenities
- developer_description
- location_description

### Character Limit Enforcement

The system enforces limits at two levels:

1. **AI Generation:** Prompts include max character counts
2. **QA Validation:** Post-generation check against template specs

### Language Requirements

- **EN:** Always required
- **AR:** Required for UAE projects
- **RU:** Required for all projects (Russian investor market)

## Maintenance

### Adding New Templates

1. Create new tab in PDP Master sheet
2. Follow column structure (Guidelines, Fields, EN, AR, RU)
3. Document in this reference file
4. Add tab name to `TEMPLATE_TABS` config
5. Create corresponding database schema if needed

### Updating Existing Templates

1. Edit directly in Google Sheets
2. Changes take effect immediately (API reads live)
3. Document significant changes in this file
4. Test with sample content generation

### Template Version Control

Google Sheets maintains version history. For major changes:
1. Use Sheets > File > Version history
2. Name significant versions
3. Note version in this document's changelog
