ARCHITECTURE HIGHLIGHTS
================================================================================

Service Boundaries:
  Client -> API Routes -> JobManager -> JobRepository -> Database
  TaskQueue -> Cloud Tasks -> Internal Endpoint -> Worker -> JobManager

State Machine:
  create_job() -> PENDING
  start_job() -> PROCESSING (enqueue to Cloud Tasks)
  complete_job() -> COMPLETED
  fail_job() -> FAILED (after 3 retries) or retry
  cancel_job() -> CANCELLED

Processing Steps (10):
  1. Upload & Validation (5%)      6. Floor Plan Extraction (60%)
  2. Image Extraction (15%)        7. Image Optimization (70%)
  3. Image Classification (30%)    8. Asset Packaging (85%)
  4. Watermark Detection (40%)     9. Cloud Upload (95%)
  5. Watermark Removal (50%)       10. Finalization (100%)

Data Flow:
  1. User creates job (POST /jobs)
  2. Job record + 10 steps created in DB with PENDING status
  3. User uploads PDF, triggers start_job()
  4. Task enqueued to Cloud Tasks with job_id + pdf_path
  5. Worker processes job, calls update_job_progress() for each step
  6. On completion, calls complete_job() with results
  7. On failure, calls fail_job() which may retry or permanently fail
  8. User polls GET /jobs/{id}/status for real-time updates

Error Recovery:
  - Cloud Tasks automatic retry (configured in queue)
  - Application-level retry tracking with exponential backoff
  - Permanent failure after 3 attempts with detailed error messages
  - Stale job detection (>24 hours in PROCESSING)
